class WiFiCallback : public BLECharacteristicCallbacks {

  void onWrite(BLECharacteristic *pCharacteristic) {

    String chunk = String(pCharacteristic->getValue());

    if (chunk.length() > 0) {

      receivedData += chunk;

      int commaIndex =
      receivedData.indexOf(',');

      if (commaIndex > 0) {

        wifiSSID =
        receivedData.substring(
          0,
          commaIndex
        );

        wifiPassword =
        receivedData.substring(
          commaIndex + 1
        );

        wifiSSID.trim();
        wifiPassword.trim();

        Serial.println();
        Serial.println("WiFi Data Received");

        Serial.print("SSID: ");
        Serial.println(wifiSSID);

        Serial.print("PASSWORD: ");
        Serial.println(wifiPassword);

        wifiConfigured = true;
        wifiConnectedOnce = false;
        bleModeStarted = false;
        preferences.begin("wifi", false);

        preferences.putString("ssid", wifiSSID);
        preferences.putString("pass", wifiPassword);

        preferences.end();

        Serial.println("WiFi Credentials Saved");
        
        WiFi.disconnect(true);
        receivedData = "";
      }
    }
  }
};


// ================= WIFI new preference funtion=================
void loadWiFiCredentials()
{
    preferences.begin("wifi", true);

    wifiSSID = preferences.getString("ssid", "");
    wifiPassword = preferences.getString("pass", "");

    preferences.end();

    if(wifiSSID.length() > 0)
    {
        wifiConfigured = true;

        Serial.println("Saved WiFi Credentials Found");
        Serial.print("SSID: ");
        Serial.println(wifiSSID);
    }
}

void setup_wifi()
{
    Serial.println("Connecting WiFi...");

    WiFi.begin(
        wifiSSID.c_str(),
        wifiPassword.c_str()
    );

    int attempts = 0;

    while(
        WiFi.status() != WL_CONNECTED &&
        attempts < 20
    )
    {
        delay(500);
        Serial.print(".");
        attempts++;
    }

    if(WiFi.status() == WL_CONNECTED)
    {
        Serial.println();
        Serial.println("Connected");
        Serial.println(WiFi.localIP());
    }
    else
{
    Serial.println();
    Serial.println("WiFi Failed");

    wifiConfigured = false;
    wifiConnectedOnce = false;

    setupBLE();
}
}
 // ota function
 void performOTA()
{
    if(WiFi.status() != WL_CONNECTED)
    {
        Serial.println("WiFi Not Connected");
        return;
    }

    Serial.println("Starting OTA Update...");

    WiFiClient otaClient;

    t_httpUpdate_return ret =
    httpUpdate.update(
        otaClient,
        OTA_URL
    );

    switch(ret)
    {
        case HTTP_UPDATE_FAILED:

            Serial.printf(
                "Update Failed (%d): %\n",
                httpUpdate.getLastError(),
                httpUpdate.getLastErrorString().c_str()
            );

            break;

        case HTTP_UPDATE_NO_UPDATES:

            Serial.println(
                "No Updates Available"
            );

            break;

        case HTTP_UPDATE_OK:

            Serial.println(
                "Update Successful"
            );

            break;
    }
}

For a **college/project demo or small pilot deployment**, this code is fairly good. It has:

✅ BLE WiFi provisioning
✅ MQTT communication
✅ OTA updates from GitHub
✅ Duplicate command prevention
✅ WiFi credential storage using Preferences
✅ Device acknowledgements
✅ Status reporting
✅ Manual WiFi reset button

But for a **real production-level IoT system (100+ devices in the field)**, I would rate it around **6/10**.

### Good things

1. BLE provisioning works.
2. OTA update mechanism exists.
3. Commands have `command_id`.
4. ACK messages are sent.
5. WiFi credentials survive reboot.
6. Device publishes status periodically.
7. MQTT reconnect exists.

---

# Major Production Issues

## 1. Infinite MQTT Reconnect Loop

Your reconnect function:

```cpp
while(!client.connected())
{
   ...
}
```

If MQTT broker goes down, ESP32 gets stuck forever.

Production systems usually do:

```cpp
if(millis()-lastReconnectAttempt > 5000)
{
   reconnect();
}
```

without blocking.

---

## 2. No Watchdog Recovery

If BLE crashes.

If OTA hangs.

If MQTT locks.

ESP32 may freeze forever.

Production devices usually enable:

```cpp
esp_task_wdt.h
```

and periodically feed watchdog.

---

## 3. Duplicate Command Protection Lost After Reboot

Currently:

```cpp
String lastCommandId = "";
```

stored in RAM only.

After reboot:

```cpp
lastCommandId=""
```

same command may execute again.

Production systems store:

```cpp
preferences.putString("lastcmd", commandId);
```

or maintain sequence numbers.

---

## 4. No Device Authentication

Anyone can publish to:

```cpp
iot/smarttank/actions
```

through HiveMQ.

They can control:

```cpp
Left_light_on
Left_fan_on
Left_pump_on
```

without permission.

Production requires:

* Username/password
* TLS MQTT
* Device tokens
* Device-specific topics

Example:

```cpp
iot/device001/actions
```

instead of

```cpp
iot/smarttank/actions
```

---

## 5. OTA Security Weak

You use:

```cpp
otaClient.setInsecure();
```

This disables certificate verification.

A fake server could deliver malicious firmware.

Production should:

```cpp
otaClient.setCACert(...)
```

and verify certificates.

---

## 6. No OTA Rollback Strategy

If firmware is bad:

```cpp
performOTA();
```

device updates.

If new firmware crashes:

Device may become unusable.

Production systems use:

* Partition A
* Partition B
* Rollback

ESP32 supports this.

---

## 7. MQTT Messages Not Retained

You publish:

```cpp
client.publish(...)
```

without checking success.

Production:

```cpp
bool ok = client.publish(...);

if(!ok)
{
   queue message
}
```

---

## 8. No WiFi Recovery Logic

If WiFi disconnects later:

```cpp
wifiConnectedOnce=true
```

stays true.

Code doesn't aggressively reconnect WiFi.

Production devices continuously monitor:

```cpp
WiFi.status()
```

and reconnect automatically.

---

## 9. Memory Fragmentation

Lots of:

```cpp
String
```

objects:

```cpp
String msg
String command
String commandId
String deviceTime
```

Long-running devices can suffer heap fragmentation.

Production systems prefer:

```cpp
char buffers[]
```

where possible.

---

## 10. BLE Can Be Started Multiple Times

You call:

```cpp
setupBLE();
```

from several places.

Repeated calls can create resource issues.

Need:

```cpp
if(!bleModeStarted)
{
   setupBLE();
   bleModeStarted=true;
}
```

---

# Scalability Problems

Current design:

```cpp
ACTION_TOPIC = "iot/smarttank/actions"
```

All devices listen to same topic.

If you deploy 100 devices:

every ESP32 receives every command.

Production:

```cpp
iot/device001/action
iot/device002/action
iot/device003/action
```

or

```cpp
iot/<device_id>/action
```

---
